🔴 Bugs Críticos — [object Promise] e contaminação de dados
Causas diretas dos erros mais frequentes reportados. Corrigir primeiro.
1. _renderizarPainel chama t.fn() inline para todas as abas CRÍTICO
Linha 120: const conteudo = t.id === abaAtiva ? t.fn() : ""
Funções async (htmlProfTurmas, htmlAdmHorarios) retornam uma Promise, não uma string. O HTML recebe "[object Promise]". É a raiz do bug que continua voltando.
Todas as seções nascem vazias (data-loaded="0"). Após inserir o HTML do painel, chamar _carregarAba(abaAtiva) que detecta automaticamente se a função é async e usa .then() ou atribui diretamente. Nunca mais chamar t.fn() inline.

3. Quatro chamadas síncronas a htmlProfTurmas() em gestao-professor.js CRÍTICO
Linhas 408, 495, 501, 510: el.innerHTML = htmlProfTurmas()
Usadas em addHorario(), delHorario(), delTurma() e addDiscNaTurma(). Todas produzem [object Promise] quando o usuário interage com essas funções.
Substituir todas por: htmlProfTurmas().then(h => { if(el) el.innerHTML = h; }) Extrair um helper _recarregarMinhasTurmas() para não repetir o padrão.

4. turmas.js — seed com 9 entradas sem profUid contamina diários CRÍTICO
As entradas 1D_GEO, 1E_GEO, 2A_GEO, 2B_GEO, 2C_GEO, 2A_SOC, 2B_SOC, 2C_SOC, 3B_ATUAL não têm profUid. O boot.js carrega este seed em RT_TURMAS para o admin. Quando salvarTudo() é chamado logado como admin, essas turmas são gravadas no Firestore diario/global sem profUid, e quando um professor faz login e o sistema sincroniza, essas turmas "escapam" para o diário dele.
Adicionar "profUid": "global" em todas as 9 entradas sem profUid no seed. Isso garante que o filtro t.profUid === uid as rejeite corretamente para professores.

🟠 Arquitetura de Dados — fontes conflitantes e filtros incompletos
Causas dos dados errados aparecendo para professores errados.
5. onSnapshot em db.js aplica RT_TURMAS sem filtrar por profUid ALTO
O listener do Firestore (db.js) aplica RT_TURMAS = rt diretamente quando chega um snapshot, sem verificar se o usuário logado é professor. Se o snapshot traz turmas sem profUid (do seed), elas entram em memória sem passar pelo filtro do boot.js. Isso reabre a contaminação toda vez que outro dispositivo/aba salva.
No bloco onSnapshot de _ativarListenerFirestore(), aplicar o mesmo filtro do boot: se professor, filtrar rt.filter(t => t.profUid === uid) antes de atribuir a RT_TURMAS.

6. _carregarHorariosGlobais — guarda só 1 entrada por slot, descarta as demais ALTO
Linha 41-42: if (!cache[chave]) { cache[chave] = {...} }
Se dois professores têm aulas diferentes na mesma hora (em turmas diferentes), apenas o primeiro é guardado. O segundo some da grade de horários. A grade "Ver Todas" e a grade do professor ficam incompletas ou mostram o professor errado.
Mudar a estrutura do cache para cache[chave] ser um array em vez de um objeto único. Atualizar todos os consumidores (_renderGradeTurma, admHCarregarGrade) para iterar o array.

7. admHVerTodas — deduplicação por ID em vez de por serie|turma|disciplina, e carrega diários duas vezes ALTO
Problema A: A deduplicação usa turmaIdsProfs.has(t.id). Turmas do seed têm IDs diferentes (ex: 1A_D1_glob) das turmas do professor (ex: 1AGEO_Bct_123), então ambas aparecem — causando tabelas duplicadas.

Problema B: Os diários dos professores são carregados duas vezes: uma para montar chavesProfSet e outra para chamar addEntrada(). Dobra o número de leituras no Firestore.
A: Deduplicar por "serie|turma|disciplina" (não por ID).
B: Carregar os diários uma única vez, armazenando em um mapa local, e usar esse mapa tanto para montar o set de deduplicação quanto para addEntrada.
